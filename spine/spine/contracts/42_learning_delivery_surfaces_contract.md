# Learning Delivery Surfaces Contract

This contract defines how learning is delivered across surfaces without fragmentation or loss of meaning.

Version: 1.0  
Effective: 2024-12-13

---

## 1. Supported Surfaces

Learning may be delivered via:

### Text
- Text is a supported delivery surface
- Text provides basic information delivery
- Text is appropriate for simple learning
- Text delivery is allowed

### Diagrams
- Diagrams are a supported delivery surface
- Diagrams provide visual information
- Diagrams are appropriate for structured learning
- Diagram delivery is allowed

### Desktop Interfaces
- Desktop interfaces are a supported delivery surface
- Desktop provides interactive visual learning
- Desktop is appropriate for complex learning
- Desktop interface delivery is allowed

### WebXR
- WebXR is a supported delivery surface
- WebXR provides web-based immersive experiences
- WebXR is appropriate for spatial learning
- WebXR delivery is allowed

### VR
- VR is a supported delivery surface
- VR provides immersive virtual experiences
- VR is appropriate for embodied learning
- VR delivery is allowed

### Spatial Computing (e.g. Vision Pro)
- Spatial computing is a supported delivery surface
- Spatial computing provides spatial understanding
- Spatial computing is appropriate for spatial learning
- Spatial computing delivery is allowed

### No Surface Is Mandatory

- No specific surface is required
- Surface selection is flexible
- Mandatory surface requirements are prohibited
- Requiring specific surfaces is a violation

---

## 2. Surface Selection Rule

Omega must choose:

### The Lightest Surface That Preserves Meaning
- Lightest surface that preserves meaning is selected
- Meaning preservation is primary
- Lightest effective surface is preferred
- Using heavier surfaces unnecessarily is prohibited

### The Least Immersive Option That Achieves Clarity
- Least immersive option that achieves clarity is selected
- Clarity achievement is primary
- Least immersive effective option is preferred
- Using more immersive options unnecessarily is prohibited

### Escalation to Immersive Surfaces Requires Justification

- Escalation to immersive surfaces requires justification
- Justification must explain why immersive is necessary
- Unjustified escalation is prohibited
- Escalating without justification is a violation

---

## 3. Surface Neutrality

Learning content must:

### Remain Identical in Intent Across Surfaces
- Learning intent is identical across all surfaces
- Intent does not change with surface
- Intent consistency is mandatory
- Changing intent across surfaces is prohibited

### Not Rely on Surface-Specific Gimmicks
- Learning does not rely on surface-specific gimmicks
- No features that only work on one surface
- Avoiding surface-specific gimmicks is mandatory
- Using surface-specific gimmicks is prohibited

### Not Hide Information Behind Immersion

- Learning does not hide information behind immersion
- All information is accessible without immersion
- Hiding information is prohibited
- Hiding information behind immersion is a violation

---

## 4. XR-Specific Constraints

When using XR:

### No Forced Movement
- XR learning does not force movement
- No walking or locomotion requirements
- Avoiding forced movement is mandatory
- Forcing movement is prohibited

### No Time Pressure
- XR learning does not create time pressure
- No deadlines or urgency
- Avoiding time pressure is mandatory
- Creating time pressure is prohibited

### No Sensory Overload
- XR learning avoids sensory overload
- Visual and auditory elements are minimal
- Avoiding sensory overload is mandatory
- Creating sensory overload is prohibited

### No Attention Traps
- XR learning avoids attention traps
- No forced focus or gaze requirements
- Avoiding attention traps is mandatory
- Creating attention traps is prohibited

### Seated Use Must Be Sufficient

- XR learning must work while seated
- Seated use is sufficient for all learning
- Requiring standing or movement is prohibited
- Requiring non-seated use is a violation

---

## 5. Persistence Rules

Learning progress must:

### Persist Across Surfaces
- Learning progress persists when switching surfaces
- Progress is maintained across surface changes
- Persistence across surfaces is mandatory
- Losing progress when switching surfaces is prohibited

### Not Reset When Switching Surfaces
- Learning progress does not reset when switching surfaces
- Progress is preserved during surface transitions
- Avoiding reset is mandatory
- Resetting progress is prohibited

### Remain Resumable Without Loss

- Learning remains resumable without loss
- Resumption preserves all progress
- Resumability without loss is mandatory
- Losing progress on resumption is prohibited

---

## 6. Accessibility

All learning must:

### Have a Non-XR Equivalent
- All learning has a non-XR equivalent
- Non-XR version is always available
- Non-XR equivalent is mandatory
- Missing non-XR equivalent is a violation

### Be Usable Without Spatial Hardware
- All learning is usable without spatial hardware
- Spatial hardware is optional, not required
- Usability without spatial hardware is mandatory
- Requiring spatial hardware is prohibited

### Respect ND-Friendly Defaults

- All learning respects ND-friendly defaults
- Neurodivergent-friendly design is standard
- ND-friendly defaults are mandatory
- Violating ND-friendly defaults is prohibited

---

## 7. Exit Safety

Leaving a surface:

### Must Be Immediate
- Exit from surface is immediate
- No delays or waiting periods
- Immediate exit is mandatory
- Delayed exit is prohibited

### Must Not Trigger Re-Engagement
- Exit does not trigger re-engagement prompts
- No follow-up messages or reminders
- Avoiding re-engagement triggers is mandatory
- Triggering re-engagement is prohibited

### Must Preserve State

- Exit preserves learning state
- State is saved and recoverable
- State preservation is mandatory
- Losing state on exit is prohibited

---

## 8. Instrumentation Limits

Surface telemetry may:

### Measure Technical Performance
- Telemetry may measure technical performance
- Performance metrics are allowed
- Technical performance measurement is appropriate
- Performance measurement is allowed

Surface telemetry may not:

### Infer Emotional State
- Telemetry does not infer emotional state
- No emotional state inference
- Inferring emotional state is prohibited
- Inferring emotional state is a violation

### Track Gaze for Optimization
- Telemetry does not track gaze for optimization
- No gaze-based optimization
- Gaze tracking for optimization is prohibited
- Gaze tracking for optimization is a violation

### Optimise Immersion Intensity

- Telemetry does not optimize immersion intensity
- No immersion intensity optimization
- Optimizing immersion intensity is prohibited
- Optimizing immersion intensity is a violation

---

## 9. Human Override

Human may:

### Lock Preferred Surfaces
- Human can lock preferred surfaces
- Surface locks prevent automatic changes
- Surface lock authority is absolute
- Surface locks cannot be overridden

### Disable Immersive Delivery
- Human can disable immersive delivery
- Immersive delivery can be turned off
- Disable authority is absolute
- Disabling immersive delivery cannot be prevented

### Request Text-Only Mode

- Human can request text-only mode
- Text-only mode removes all non-text elements
- Text-only mode request authority is absolute
- Text-only mode requests cannot be denied

---

## 10. Immutability

### This Contract Changes Only via Explicit Revision

- Contract changes require explicit revision
- No ad-hoc edits to surface delivery rules or constraints
- Explicit revision is mandatory
- Ad-hoc changes are prohibited

### Revision Requirements

- Contract revisions must be documented
- Revisions require version control
- Revisions are traceable and auditable
- Undocumented revisions are violations

---

## Change Control

This contract changes only via explicit revision:

### Explicit Revision
- Contract changes must be explicit and documented
- No ad-hoc edits to surface delivery principles or rules
- Changes require version control
- Changes are traceable and auditable

### Documented Rationale
- Every contract change must include rationale:
  - What changed in the contract
  - Why it changed
  - What it affects
  - Expected impact on learning delivery and accessibility

### Rollback Plan
- Every contract change must have rollback plan
- Previous contract versions must be preserved
- Rollback procedure must be documented
- Rollback can be executed at any time

---

## Enforcement

Violations of this contract include:
- Requiring specific surfaces
- Escalating to immersive surfaces without justification
- Changing learning intent across surfaces
- Using surface-specific gimmicks
- Hiding information behind immersion
- Violating XR-specific constraints
- Losing progress when switching surfaces
- Missing non-XR equivalents
- Inferring emotional state or optimizing immersion
- Missing or incomplete exit safety

Violations must be:
- Logged as incidents
- Reviewed by system owner
- Corrected immediately
- Documented in change logs

---

## Amendments

Changes to supported surfaces, selection rules, surface neutrality, XR constraints, or accessibility requirements require:
- Impact assessment on learning delivery and accessibility
- Testing with representative learning scenarios
- Approval from system owner
- Version increment
- Documentation in change logs
